Adaptive Fitness Coach / Watch

The watch is the whole product in-workout. Every screen below is a real Apple workout session (WorkoutKit/HealthKit) with a guidance layer on top. The OS is the system of record — the app records nothing and the user logs nothing.

In-workout — the two interval states

The core loop. The user follows the buzz: run when it says run, walk when it says walk. These screens are confirmation, not instruction — they should be readable in a glance but never required.

Behavior
  • One state fills the screen: a verb (RUN / WALK), a direction glyph, and the current interval's elapsed timer.
  • Interval length is not fixed — it is recomputed live from heart rate (see A4). The timer counts the live interval, it is not a script playing out.
  • Color is semantic and consistent everywhere: green = work, amber = recover. Carry these into haptics and notifications too.
  • HR + zone bar are ambient reassurance. Nothing here is tappable mid-run; the user is moving.
Data / API
  • Live HR from the active HKWorkoutSession
  • Zone computed app-side from raw HR + user max HR (do not depend on the OS exposing a live zone — Q3)
  • Pace/route captured natively; app reads, never writes
Haptics
  • → Run: distinct double tap, sharp.
  • → Walk: single long, soft.
  • Must be feelable mid-stride without looking (N5).

A4

Adaptation moment — HR runs hot

The product's thesis made visible: the plan bends to the body, silently. Sustained high HR → the app shortens the run and lengthens the next walk. How much it announces this is the open dial (Q5) — default is one calm line, no nagging.

♥ 171 ↑7:21
WALK
Easing off —
longer recovery
HR high · auto-adjusted
Control logic (binding on engineering)
  • Act on smoothed / sustained HR over a window, never a single spot reading — HR lags effort by 30–60s+, more for a deconditioned user.
  • Bias toward backing off. "End the run early / lengthen the walk" needs a shorter confirming window than "extend the run," which is the higher-risk action and requires a longer one.
  • If HR is not recovering during the walk, lengthen the walk rather than start the next run.
  • Adaptation works interval-to-interval within the session; mid-interval throttling is a later refinement, not required for P0.
Surfacing (Q5 — design dial)
  • Default: one calm line + the normal switch haptic. No alert, no modal, no "why".
  • The change must feel like the session breathing with the user — not the app correcting an error.

Exercise card — first time vs learned

A strength day is an ordered sequence of cards. Early on, each card carries a brief form demo; once the user knows the movement, the demo collapses and the card gets compact. The app proposes a weight — a forward-looking seed, never a log.

Behavior
  • Cards advance in sequence; progress dots show position in the session.
  • Form demo is shown by default for the first N exposures of a movement, then auto-collapses to the compact card. User can re-show it anytime.
  • Suggested weight is an app-proposed conservative default within the user's dumbbell range, adjustable with ± . This is a seed, not a log — adjusting it does not "record" anything (N1/N7).
  • P1 is static: no session-to-session progression yet. The weight is the same conservative default each session until P2 turns on adaptation.
Data / API

  • Live HR shown ambiently; not used to adapt in P1
Haptics
